# 当前软件使用手册

本文档记录 `boss-jobpilot` 当前版本的实际可用功能。后续每次迭代新增或改变用户路径时，都应同步更新本文档。

## 当前能力边界

当前版本已经支持：

- 本地启动 API 和 Web 工作台。
- 维护个人经历库。
- 手动保存岗位。
- 使用 Chrome 插件在 Boss 直聘岗位页一键采集当前岗位。
- 使用 Chrome 插件在 Boss 直聘岗位页读取本地最新打招呼语，并复制或填入页面输入框。
- 对岗位进行规则版匹配分析。
- 将岗位分析结果保存为历史记录。
- 基于最新岗位分析和匹配经历生成 Markdown 定制简历草稿。
- 支持生成快投版、正式版和技术版三种 Markdown 定制简历草稿。
- 基于最新岗位分析和匹配经历生成礼貌版、证据版和主动版打招呼语草稿。
- 一键复制 Markdown 简历草稿和打招呼语草稿。
- 一键复制完整投递包 Markdown 文本。
- 在同一岗位下查看最新和上一版简历、打招呼语的轻量对比。
- 在岗位池中按投递状态查看数量并筛选岗位。
- 在岗位卡片中手动标记投递状态：已打招呼、已投递、已回复、面试、Offer、已拒绝、关闭。
- 在岗位卡片中查看投递状态变更时间线。
- 在岗位卡片中设置下次跟进时间，并通过“今日待跟进”入口筛选今天及以前需要处理的岗位。
- 在岗位池中查看第一版投递复盘摘要，包括投递推进率、回复率、面试转化、平均匹配分、跟进风险、策略建议、AI 策略复盘、轻量分布和效果归因。
- 在投递复盘区查看最近 AI 生成记录，并按能力、状态和岗位筛选，定位 Provider 成功、降级或规则版调用。

当前版本尚未支持：

- 自动打招呼或自动投递。
- 自动批量浏览 Boss 岗位列表。
- 简历在线编辑器、完整 diff 和导出 PDF/DOCX。
- 自动发送打招呼语。
- 自动提醒推送和面试准备闭环。

## 一键启动

Windows 新手用户可以在项目根目录双击：

```text
start.bat
```

脚本会检查 Node/Corepack，必要时安装依赖，并启动：

- API：`http://127.0.0.1:4000`
- Web：`http://127.0.0.1:5173`

关闭项目时，关闭 `boss-jobpilot API` 和 `boss-jobpilot Web` 两个命令行窗口即可。

## 程序员启动

在项目根目录运行：

```powershell
npm run dev
```

该命令只启动 API 和 Web，不启动插件。

单独启动：

```powershell
npm run dev:api
npm run dev:web
```

插件开发启动：

```powershell
npm run dev:extension
```

完整验证：

```powershell
npm run typecheck
npm run test
npm run lint
npm run build
```

## 配置 PackyAPI

项目已经预留 OpenAI-compatible AI Provider，默认面向 PackyAPI。

PowerShell 示例：

```powershell
$env:AI_API_KEY="你的 PackyAPI Key"
$env:AI_API_BASE_URL="https://www.packyapi.com/v1"
$env:AI_MODEL="gpt-5"
npm run dev
```

也可以复制 `.env.example` 为 `.env` 后填写：

```text
AI_PROVIDER=packyapi
AI_API_KEY=你的 PackyAPI Key
AI_API_BASE_URL=https://www.packyapi.com/v1
AI_MODEL=gpt-5
```

配置 `AI_API_KEY` 后，岗位分析、定制简历、打招呼语和 AI 策略复盘会优先使用 Provider；不配置时，软件会继续使用本地规则版生成逻辑。Provider 调用失败时，本次操作会回退到规则版结果并在页面提示降级原因。不要把真实密钥提交到 Git。

Web 工作台的投递复盘区域会显示 AI Provider 状态：未配置、可用或验证失败。点击“刷新”会让本地 API 发起一次最小 JSON 探测，用来确认 PackyAPI 密钥、模型和网络是否可用；探测不会发送你的简历、经历库或岗位 JD。

投递复盘区域还会显示最近 AI 生成记录，包括岗位分析、定制简历、打招呼语和策略复盘的运行状态、Provider、耗时、Prompt 版本和降级原因。可以按能力、状态和关联岗位筛选，并点击记录查看明细。这里不会保存完整 Prompt、简历正文、经历库正文或 JD 正文。

## 使用 Web 工作台

打开：

```text
http://127.0.0.1:5173
```

### 维护经历库

在经历库中录入你的项目、实习、工作、开源或教育经历。建议尽量填写：

- 标题：项目或经历名称。
- 组织：公司、学校、团队或项目来源。
- 角色：你在经历中的角色。
- 技术栈：用于岗位匹配。
- 职责：你具体做了什么。
- 成果：交付结果或业务影响。
- 指标：量化结果。
- 证据等级：是否能深聊，或是否不能使用。
- 负责程度：主导、负责、参与或协助。

`do_not_use` 的经历不会参与岗位匹配和简历生成。

经历库右侧列表上方提供筛选入口：

- 搜索经历：可按标题、组织、角色、摘要、职责、成果、技术栈、指标和标签搜索。
- 技能筛选：点击技能标签，只显示包含该技术栈的经历。
- 标签筛选：点击标签，只显示包含该标签的经历。
- 清除筛选：恢复显示全部经历素材。

### 保存岗位

可以在 Web 的岗位采集表单中手动填写：

- 平台。
- 城市。
- 岗位标题。
- 公司。
- 薪资。
- 经验要求。
- 学历要求。
- 链接。
- JD 原文。

点击“保存岗位”后，岗位会进入岗位池。

在岗位卡片右上角点击删除图标，可以删除已保存岗位。删除岗位会同时清理该岗位关联的分析记录、简历版本、打招呼语草稿和投递状态事件。

### 使用投递看板

岗位列表上方会显示投递状态看板：

- 全部。
- 未生成草稿。
- 草稿。
- 已打招呼。
- 已投递。
- 已回复。
- 面试中。
- Offer。
- 已拒绝。
- 已关闭。

点击任一状态会筛选岗位列表，只显示该状态下的岗位。再次点击“全部”可回到完整岗位池。

看板下方还会显示跟进队列：

- 今日待跟进：逾期和今天到期的岗位。
- 逾期：下次跟进时间早于今天零点的岗位。
- 今天：下次跟进时间在今天内的岗位。
- 未来 3 天：下次跟进时间在今天之后、未来 3 天内的岗位。

点击任一队列会筛选岗位列表，只显示该队列下需要处理的岗位。

### 查看投递复盘

岗位池顶部会显示“投递复盘”区域，数据会随着当前保存的岗位、岗位分析、简历版本和投递状态自动更新。

复盘区域顶部可以筛选复盘范围：

- 状态：全部状态、未生成草稿、草稿、已打招呼、已投递、已回复、面试中、Offer、已拒绝、已关闭。
- 投递建议：全部建议、未分析、优先投递、可以投递、谨慎投递、建议跳过。
- 城市：按当前已保存岗位中的城市筛选，未填写城市会单独显示。

筛选后，投递推进率、回复率、面试转化、轻量分布、效果归因和策略建议都会基于当前复盘范围重新计算；岗位列表本身不会被这个复盘筛选影响。

当前复盘指标包括：

- 投递推进率：已进入“已投递、已回复、面试中、Offer、已拒绝、已关闭”的岗位数 / 已生成话术的岗位数；如果还没有话术，则使用全部岗位数作为参考分母。
- 回复率：已回复、面试中或 Offer 的岗位数 / 已投递及以后状态的岗位数。
- 面试转化：面试中或 Offer 的岗位数 / 已投递及以后状态的岗位数。
- 平均匹配分：所有已分析岗位的平均匹配分。
- 跟进风险：提示逾期跟进数量，以及推进中但没有设置下次跟进的岗位数量。

复盘区域还会展示三组轻量分布：

- 投递建议分布：优先投递、可以投递、谨慎投递、建议跳过。
- 城市分布：当前岗位池内岗位的城市分布。
- 版本迭代：未生成、1 版、2 版以上，用于快速判断哪些岗位已经做过简历或话术迭代。

复盘区域还会展示“效果归因”，用于快速判断哪些投递方向更值得继续投入：

- 岗位类型：根据岗位标题轻量归类为 AI / 算法、全栈 / 后端、前端、数据、实习 / 校招或其他。
- 公司类型：优先根据公司名轻量归类为科技产品、金融、外包 / 服务商、教育、电商 / 零售、游戏、其他公司；公司名缺失时会回退使用岗位标题和 JD。
- 城市：按岗位城市分组。
- 投递建议：按岗位分析中的优先投递、可以投递、谨慎投递、建议跳过分组。
- 简历版本：按打招呼语关联的具体简历版本 ID 分组；未关联简历版本和未生成话术会单独显示。

每个归因分组会展示该组的已投递数量、总岗位数量、回复率和面试率。当前归因仍是轻量统计，适合发现趋势，不等同于严格因果分析。

复盘区域还会展示“策略建议”。当前建议是规则版，会根据逾期跟进、未设置下次跟进、回复率、平均匹配分、定制简历覆盖率和归因中的有效信号生成最多 4 条行动建议。它可以作为每日投递前的检查清单。

复盘区域还会自动请求本地 API 生成“AI 策略复盘”。当前版本会输出复盘摘要、下一步重点、建议实验和风险提醒；配置 PackyAPI 后可使用真实模型，未配置密钥时模型名仍显示为 `rule-based`。复盘区顶部的 AI Provider 状态条可用于正式投递前检查密钥和网络状态。

“最近 AI 生成”会展示最近 AI 相关操作。可以使用能力、状态和岗位筛选记录；点击任一记录后，明细面板会展示能力、状态、关联岗位、Provider、模型、Prompt 版本、耗时、时间和错误摘要。如果状态是“已降级”，说明真实 Provider 调用失败，本次结果已自动使用规则版兜底。

### 分析岗位

在岗位卡片右上角点击图表图标。

系统会生成并保存：

- 匹配分数。
- 投递建议。
- 匹配关键词。
- 必需技能。
- 加分技能。
- 匹配经历标题、摘要和技术栈。
- 风险信号。
- 简历策略。

### 生成定制简历

在已经分析过的岗位卡片右上角点击简历版本按钮：

- “快”：生成快投版，内容更短，适合快速粘贴或作为一页简历基础。
- “正”：生成正式版，内容更完整，适合正式投递。
- “技”：生成技术版，突出技术栈、职责和可面试追问的实现证据。

系统会基于：

- 当前岗位信息。
- 最新岗位分析。
- 匹配到的经历。

生成 Markdown 简历草稿，并保存为该岗位的简历版本。岗位卡片会显示最新生成的版本：

- 定制说明。
- 变更摘要。
- Markdown 内容预览。

点击简历草稿面板右上角的“复制”按钮，可以把完整 Markdown 简历复制到剪贴板。

如果需要微调 AI 或规则版生成结果，可以在“编辑 Markdown 简历”文本框中直接修改内容，然后点击“保存为新版本”。系统不会覆盖原版本，而是创建一个 `edited-*` 简历版本，版本对比和投递包会自动使用最新版本；如果该岗位已有投递草稿，草稿也会自动关联到这份新简历版本。

如果还没有分析岗位，系统会提示先进行岗位分析。

如果同一岗位生成过多版简历，岗位卡片底部会出现“版本对比”。当前版本会对比最新简历和上一版简历，展示生成时间、版本类型、变更摘要、正文长度和使用经历数量，并支持复制任一版本。

### 生成打招呼语

在已经分析过的岗位卡片右上角点击打招呼语版本按钮：

- “礼”：生成礼貌版，语气更克制，适合不确定岗位细节或想降低打扰感时使用。
- “证”：生成证据版，默认版本，突出最匹配的项目经历和技能证据。
- “直”：生成主动版，更明确表达进一步沟通意愿，适合高匹配岗位。

系统会基于：

- 当前岗位信息。
- 最新岗位分析。
- 匹配到的经历。
- 最新定制简历版本，如果已经生成。

生成一条短打招呼语，并保存为该岗位的 `application draft`。岗位卡片会显示：

- 打招呼语草稿。
- 话术版本。
- 草稿状态。
- 已关联的简历版本 ID，如果生成时已有简历版本。

点击打招呼语草稿面板右上角的“复制”按钮，可以把话术复制到剪贴板。

如果需要微调话术，可以在“编辑打招呼语”文本框中修改内容，然后点击“保存话术”。保存会更新当前 `application draft`，不会自动发送给招聘者。

当前版本只生成草稿，不会自动发送给招聘者。

如果同一岗位生成过多版打招呼语，“版本对比”会展示最新话术和上一版话术的版本、状态、更新时间和消息摘要，并支持复制任一版本。

### 复制完整投递包

在岗位卡片右上角点击复制图标，可以把当前岗位的完整投递包复制到剪贴板。

投递包会按 Markdown 组织当前已有内容：

- 岗位信息和 JD。
- 岗位分析、匹配分、技能和风险信号。
- 匹配经历详情。
- 定制简历元信息和 Markdown 简历正文。
- 打招呼语、投递状态和状态时间线。

如果某部分尚未生成，投递包会保留明确占位说明，便于后续补齐。

### 标记投递状态

生成打招呼语后，岗位卡片的打招呼语草稿面板会出现状态按钮：

- 已打招呼。
- 已投递。
- 已回复。
- 面试。
- Offer。
- 已拒绝。
- 关闭。

点击按钮只会更新本地投递记录，不会操作 Boss 页面，也不会自动发送消息。

当状态从一个阶段切换到另一个阶段时，系统会在本地 `application_events` 中记录一次状态变更事件。标记为“已投递”时，如果还没有投递时间，系统会自动写入当前时间。

状态变更后，打招呼语草稿面板底部会出现“状态时间线”。时间线会按发生时间展示每次状态变化，例如从“草稿”更新为“已打招呼”、从“已投递”更新为“面试中”。

### 设置跟进时间

生成打招呼语后，岗位卡片的打招呼语草稿面板会出现“下次跟进”时间输入框。

选择日期和时间后，点击“保存”，系统会写入本地投递记录。到期当天及逾期未处理的岗位，会出现在岗位池上方的跟进队列中。

如果不再需要跟进，可以点击时间输入框旁边的“清除”按钮。

## 使用 Chrome 插件采集 Boss 岗位

先确保本地 API 已启动：

```text
http://127.0.0.1:4000
```

构建或启动插件：

```powershell
npm run dev:extension
```

或生产构建：

```powershell
corepack pnpm --filter @boss-jobpilot/extension build
```

打开 Chrome：

```text
chrome://extensions
```

开启“开发者模式”，点击“加载已解压的扩展程序”。

开发模式通常加载：

```text
D:\projects\boss\apps\extension\build\chrome-mv3-dev
```

生产构建加载：

```text
D:\projects\boss\apps\extension\build\chrome-mv3-prod
```

打开 Boss 直聘岗位详情页：

```text
https://www.zhipin.com/...
```

点击 Chrome 右上角 `Boss Jobpilot` 插件图标，再点击“保存当前岗位”。保存成功后，回到 Web 工作台刷新岗位池即可看到岗位。

如果当前岗位已经保存过，插件会提示“岗位已在本地岗位池中，无需重复保存”，不会再次写入一条重复岗位。重复识别优先使用去掉查询参数和锚点后的稳定 URL；URL 不可用时，会使用岗位标题和公司名匹配。

如果本地 API 没有启动，插件会提示“本地 API 不可用，请先启动 boss-jobpilot”。如果当前页面不是可识别的 Boss 岗位页，插件会提示当前页面不可识别。

生成过打招呼语后，也可以在同一个 Boss 岗位页打开插件：

1. 点击“读取本地打招呼语”。
2. 插件会用当前页面 URL、岗位标题和公司名匹配本地岗位池。
3. 匹配成功后，可以点击“复制话术”复制到剪贴板。
4. 也可以点击“填入页面”，插件会尝试把话术写入当前页面可见的聊天输入框。
5. 填入后请先检查内容，再由你手动点击 Boss 页面上的发送按钮。

需要在 Boss 页面直接留档或复制完整材料时：

1. 点击“读取本地投递包”。
2. 插件会匹配当前岗位，并从本地 API 读取完整 Markdown 投递包。
3. 点击“复制投递包”，即可复制包含岗位信息、JD、分析、匹配经历、定制简历、打招呼语和状态时间线的 Markdown 文本。

如果插件提示“本地岗位池还没有匹配当前页面的岗位”，先点击“保存当前岗位”，再回到 Web 工作台完成岗位分析和打招呼语生成。

如果插件提示“没有找到可填写的聊天输入框”，说明当前页面没有打开聊天输入区域，或 Boss 页面结构发生变化。此时可以使用“复制话术”后手动粘贴。

## 推荐工作流

1. 启动 `start.bat` 或 `npm run dev`。
2. 在经历库录入所有可用经历。
3. 打开 Boss 岗位详情页，用插件保存岗位。
4. 回到 Web 岗位池，点击“分析”。
5. 查看匹配分数、风险和匹配经历。
6. 选择“快 / 正 / 技”生成对应版本的定制简历。
7. 选择“礼 / 证 / 直”生成对应版本的打招呼语。
8. 点击岗位卡片右上角的复制图标，复制完整投递包留档；需要单独投递时，也可点击草稿面板的“复制”按钮复制打招呼语或 Markdown 简历。
9. 回到 Boss 岗位页，使用插件读取本地打招呼语并复制或填入页面输入框，检查后手动发送。
10. 手动完成打招呼或投递后，在岗位卡片中标记对应状态。
11. 设置下次跟进时间。
12. 使用投递看板按状态或跟进队列检查待推进岗位。
13. 多次生成简历或打招呼语后，使用“版本对比”检查最新和上一版差异，再选择要复制的版本。

## 故障排查

### 插件点击后提示保存失败

通常是本地 API 没启动。先确认：

```text
http://127.0.0.1:4000/health
```

如果无法访问，重新运行：

```powershell
npm run dev
```

### Chrome 加载插件后没有按钮

确认加载的是构建产物目录，不是源码目录：

```text
apps/extension/build/chrome-mv3-prod
```

或开发模式目录：

```text
apps/extension/build/chrome-mv3-dev
```

### 生成简历提示需要先分析岗位

先点击岗位卡片右上角的图表图标进行分析，再点击文档图标生成简历。

### 生成打招呼语提示需要先分析岗位

先点击岗位卡片右上角的图表图标进行分析，再点击消息图标生成打招呼语。

### 匹配经历仍显示为 ID

如果岗位分析结果里的某个经历 ID 在当前经历库中找不到，系统会退回显示原始 ID。通常是该经历后来被删除或当前数据文件不是生成分析时的数据文件。

### 删除岗位提示失败

当前版本会在删除岗位时同步清理该岗位下的分析、简历、投递草稿和事件。如果仍然失败，先刷新岗位池确认该岗位是否已经被其他窗口删除，再检查本地 API 是否仍在运行。

## 维护规则

每次项目迭代后，需要同步检查并更新：

- 本文档中的功能边界。
- 启动方式。
- Web 工作流。
- 插件加载方式。
- 新增 API 或用户入口。
- 故障排查项。
